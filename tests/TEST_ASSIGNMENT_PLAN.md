## 📅 测试时间安排

### 方案 A: 集中测试（推荐）

**时间**: 连续 1-2 天  
**优势**: 快速发现问题，及时反馈

```
Day 1 (上午 9:00 - 12:00)
├─ 09:00 - 09:30  环境准备与配置验证
├─ 09:30 - 10:30  后端 API 功能测试
├─ 10:30 - 11:30  安全性测试
├─ 11:30 - 12:00  性能测试

Day 1 (下午 14:00 - 18:00)
├─ 14:00 - 15:30  前端 Web 端测试
├─ 15:30 - 16:30  Desktop Manager 测试
├─ 16:30 - 17:30  Admin 后台测试
├─ 17:30 - 18:00  数据库测试

Day 2 (上午 9:00 - 12:00)
├─ 09:00 - 10:30  Docker 容器化部署测试
├─ 10:30 - 11:30  端到端集成测试
├─ 11:30 - 12:00  文档验证测试

Day 2 (下午 14:00 - 17:00)
├─ 14:00 - 15:30  问题修复验证
├─ 15:30 - 16:30  回归测试
├─ 16:30 - 17:00  生成最终测试报告
```

### 方案 B: 分散测试

**时间**: 3-5 天，每天 2-3 小时  
**优势**: 不影响其他工作，压力较小

```
Day 1: 环境准备 + 后端 API 测试
Day 2: 安全性测试 + 性能测试
Day 3: 前端功能测试
Day 4: Docker 部署 + 数据库测试
Day 5: 集成测试 + 报告生成
```

## 🎯 测试执行流程

### 第一阶段：准备阶段（30分钟）

**负责人**: 测试负责人 + DevOps

```bash
# 1. 克隆最新代码
git clone https://github.com/MatuX-ai/OpenMtSciED.git
cd OpenMtSciED

# 2. 安装依赖
pip install -r requirements.txt
cd desktop-manager && npm install
cd ../admin-web && npm install

# 3. 配置环境变量
copy .env.example .env.local
# 编辑 .env.local 填写实际配置

# 4. 启动所有服务
# （按照 DEPLOYMENT_TEST_GUIDE.md 中的说明）
```

**验收标准**:
- [ ] 所有依赖安装成功
- [ ] .env.local 配置正确
- [ ] 后端服务启动成功（端口 8000）
- [ ] Web 前端可访问（端口 3000）
- [ ] Admin 后台可访问（端口 4200）

---

### 第二阶段：自动化测试（30-45分钟）

**负责人**: 后端测试工程师

```bash

cd tests
python quick_health_check.py

# 2. 运行完整测试套件
python deployment_pre_check.py

# 3. 查看测试结果

**预期输出**:
```

- [ ] 所有容器正常启动

### 第五阶段：问题修复与回归测试（60-90分钟）

**负责人**: 全体测试人员

1. **整理问题清单**
   - 汇总所有发现的 bug
   - 按优先级排序（P0/P1/P2/P3）
   - 创建 GitHub Issue

2. **开发团队修复**
   - 优先修复 P0 问题
   - 其次修复 P1 问题

3. **回归测试**
   - 重新运行自动化测试
   - 验证修复的问题
   - 确保没有引入新问题

---

### 第六阶段：生成测试报告（30分钟）

**负责人**: 测试负责人

1. **汇总测试结果**
   - 自动化测试报告
   - 手动测试记录
   - 性能测试数据
   - 安全性评估

2. **评估部署就绪度**
   - 计算各项评分
   - 给出明确建议

3. **编写正式报告**
   - 使用 `tests/TEST_REPORT_TEMPLATE.md` 模板
   - 包含所有关键信息
   - 附上详细数据

4. **提交报告**
   - 发送给项目负责人
   - 上传到项目文档
   - 归档测试数据

## ⚠️ 风险与应对

### 潜在风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| 测试环境问题 | 中 | 高 | 提前准备，备用环境 |
| 发现严重 bug | 中 | 高 | 预留修复时间 |
| 人手不足 | 低 | 中 | 调整测试范围 |
| 时间不够 | 低 | 中 | 优先测试核心功能 |
| 第三方服务不可用 | 低 | 中 | Mock 或跳过 |

### 应急预案

**如果发现 P0 级别严重问题**:
1. 立即停止测试
2. 通知项目负责人和开发团队
3. 评估修复时间
4. 决定是否延期部署

**如果测试环境故障**:
1. 切换到备用环境
2. 或使用 Docker 快速重建
3. 记录故障原因，避免重复
