# 项目技术栈变更通知

**变更日期**: 2026-04-18  
**变更类型**: 技术方案调整  
**影响范围**: 学习路径推荐模块

---

## 📢 变更概述

经项目组决定，**移除PPO强化学习实现方案**，改为**基于规则引擎的自适应路径调整算法**。

### 变更原因

1. **降低复杂度**: PPO实现需要大量训练数据和调优工作
2. **加快上线速度**: 规则引擎可快速实现和迭代
3. **提高可维护性**: 规则逻辑更易于理解和调试
4. **减少依赖**: 无需引入stable-baselines3等大型库
5. **成本效益**: 在初期阶段，规则引擎已能满足需求

---

## 🔄 技术方案对比

| 维度 | 原方案 (PPO) | 新方案 (规则引擎) |
|------|-------------|------------------|
| **技术栈** | Stable Baselines3 + Gymnasium | 自定义Python服务 |
| **开发工时** | 8人天 | 6人天 |
| **训练数据需求** | ≥1000条用户记录 | 无需训练数据 |
| **可解释性** | 低 (黑盒模型) | 高 (透明规则) |
| **灵活性** | 高 (自动学习) | 中 (需人工调整) |
| **维护难度** | 高 | 低 |
| **上线速度** | 慢 (需训练) | 快 (即写即用) |

## 📊 影响评估

### 对现有功能的影响
- ✅ **无破坏性变更**: 现有API接口保持不变
- ✅ **向后兼容**: 不影响已实现的路径生成功能
- ⚠️ **功能降级**: 个性化程度略低于PPO方案（但仍在可接受范围）

### 对开发团队的影响
- ✅ **降低学习曲线**: 无需掌握强化学习知识
- ✅ **简化调试流程**: 规则逻辑易于追踪
- ✅ **加快迭代速度**: 规则调整无需重新训练

### 对用户的影响
- ✅ **更快的响应速度**: 无需加载大型模型
- ✅ **更透明的推荐**: 可解释为什么推荐某条路径
- ⚠️ **个性化略低**: 但通过精细的规则设计可弥补
